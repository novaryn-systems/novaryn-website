Data Security & Processing

XPO Data Pipeline: From File to Knowledge

Understanding AxJedi's data flow is critical for security and trust. Here's exactly what happens to your code:

Processing Pipeline

Stage Input Processing Location Output Internet Transmission
1. Parse XPO Raw XPO file (400 MB) Local laptop Structured code + metadata NO
2. Chunk & Extract Parsed code Local laptop 188,901 chunks (method signatures, structure, references) NO
3. Build Dependency Graph Chunk relationships Local laptop 44,232 nodes + 23,430 edges in Postgres NO
4. Generate Embeddings Function signatures only (sys+syp layers) OpenAI API (internet) Vector embeddings (stored locally) YES - function signatures only
5. Query Processing User question Local + OpenAI API Grounded answer with code references Hybrid (see note below)

What Gets Sent to the Internet (and Why It's Safe)

During Initial Setup (One-time):

  • Function signatures from sys+syp layers only (not custom Vestergaard code)
    • Example: "public static void initFromSalesLine(SalesLine line)" (the signature)
    • NOT sent: Business logic, calculations, proprietary algorithms
  • Why this is safe: Function signatures are part of standard Ax Dynamics. They contain no secrets, no PII, no proprietary business logic. Think of it like sending an API contract.

During Daily Queries (Normal Operation):

  • User question only (e.g., "How do I fix the discount calculation?")
  • NOT sent: Any actual code, chunks, or Vestergaard data
  • Response: AI suggestions grounded in the local graph (you stay in control)

Zero Proprietary Risk

Zero proprietary Vestergaard business logic leaves localhost during normal operation. The local Postgres database is your fortress. All actual code chunk processing happens there. The internet-connected AI model only ever sees function signatures (during setup) or generic questions (during queries).

Layer Distinction: sys, syp, custom

Dynamics AX code is organized into layers. AxJedi distinguishes between them:

SYS Layer (Core)

Standard Dynamics AX platform code. Identical across all installations. Safe to expose for embeddings.

SYP Layer (Patches)

Microsoft hotfixes and updates. Also standard across installations. Safe to expose for embeddings.

VAR/Custom Layer (Vestergaard)

Proprietary business logic unique to Vestergaard. Never exposed except in fully local processing.

Example: How a Query Stays Secure

Scenario: Sune asks "How do I integrate with our proprietary invoice system?"

  1. Query stays local: Question processed against local Postgres and Sigma graph
  2. Local results: All references to Vestergaard invoice code found in local database
  3. Optional AI enhancement: If Sune wants more context, only the question and generic method signatures are sent to Claude for explanation—never the actual proprietary code
  4. Result: Expert-level answer without exposing your IP
