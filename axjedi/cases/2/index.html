Call SQL Stored Procedure from X++

Integration / External Systems⚠️ Bounced 1 time then verified

The Question

“wan't to call sql stored procedure and get result from x++”
⬇️ What you see below is the actual AxJedi response - the five expert sections exactly as delivered to the developer. Every identifier verified. Every method checked against the real codebase.

📋 Approach

Based on my investigation of the Customer codebase, AX 2009 uses the Connection class (not OdbcConnection) along with Statement and ResultSet for SQL operations. The security model uses SqlStatementExecutePermission for authorization. I found multiple examples in the codebase showing the correct pattern: create a Connection, get a Statement from it, execute SQL with proper permissions, and process results. I'll demonstrate calling a stored procedure with parameters and retrieving results using these verified classes.

216                                    DateYear::Digits4));
217                break;
218            default:
219                paramList += strfmt("'%1'", any2str(conPeek(_parameters, i)));
220        }
221    }
222
223    try
224    {
225        connection = new Connection();
226        statement = connection.createStatement();
227        
228        // Build SQL statement
229        sqlStmt = strfmt("EXEC %1 %2", _procedureName, paramList);
230        
231        // Set up security permission
232        permission = new SqlStatementExecutePermission(sqlStmt);
233        permission.assert();
234        
235        // Execute the stored procedure (for non-query operations)
236        rowsAffected = statement.executeUpdate(sqlStmt);
237        
238        CodeAccessPermission::revertAssert();
239        
240        success = true;

243    }
244    catch (Exception::Error)
245    {

💡 Learning Point

Direct SQL Execution in AX 2009

AX 2009 provides direct SQL access through the Connection/Statement/ResultSet pattern, which mirrors JDBC but with AX-specific security layers. This architecture enables stored procedure calls while maintaining the security model that protects against SQL injection and unauthorized database access.

Security Architecture: SqlStatementExecutePermission is the gatekeeper for all direct SQL operations. The permission must be asserted before executing any SQL statement and should be reverted immediately after execution. This follows the principle of least privilege - the elevated permission exists only for the minimal time needed.

Connection Management: Unlike ODBC or ADO.NET patterns, AX 2009's Connection class doesn't require explicit disposal. The AX kernel manages the underlying database connections through its connection pool. Creating a new Connection() uses the current AX session's database context, inheriting the company, user credentials, and transaction scope.

Parameter Safety: When building SQL statements with user input, always escape single quotes using strReplace(value, "'", "''"). For complex scenarios, consider building parameterized queries through the Query framework instead of direct SQL to avoid injection risks.

Transaction Context: SQL executed through Statement operates within the current AX transaction scope (ttsbegin/ttscommit). If you need isolated transactions, you must manage them explicitly in your stored procedure. Be aware that stored procedure transactions can conflict with AX's transaction management.

Result Processing: ResultSet provides typed accessors (getString, getInt, getReal, getDate) with 1-based column indexing. The next() method advances the cursor and returns false when no more rows exist. For large result sets, process data incrementally rather than loading everything into memory to avoid performance issues.
